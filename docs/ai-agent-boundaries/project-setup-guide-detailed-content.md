#### **Урок 1.1: Философия правильной организации проекта**
**Основные концепции:**
- 🎯 **Layered Architecture (Слоистая архитектура)** - разделение приложения на независимые слои
- 🎯 **Separation of Concerns** - каждая часть системы отвечает за свою область ответственности
- 🎯 **Single Responsibility Principle (SRP)** - одна функция/модуль = одна задача
- 🎯 **DRY (Don't Repeat Yourself)** - избегать дублирования кода
- 🎯 **Presentation → Business Logic → Data Access** - три основных слоя приложения

**Технологии и инструменты:**
- Layered Architecture для FastAPI и React проектов
- Routes/Services/Models паттерн для backend
- Components/Services паттерн для frontend
- Избегание "god objects" (всё в одном файле)

**Практические навыки:**
- Понимание зачем разделять код на слои
- Определение где должна находиться логика (router vs service)
- Избегание смешивания HTTP логики с бизнес-логикой
- Построение maintainable проектов с ИИ
- Правильное общение с Claude Code для создания структурированного кода

**Контекст применения:**
Объясняет философию через аналогии (ресторан: официант/повар/кладовая). Показывает разницу между плохой (всё в одном файле) и хорошей (слоистой) архитектурой. ASCII диаграммы иллюстрируют flow запросов через слои.

#### **Урок 1.2: Инициализация проекта - первые шаги**
**Основные концепции:**
- 🎯 **Git init первым делом** - система контроля версий с самого начала
- 🎯 **Virtual environment** - изоляция зависимостей проекта
- 🎯 **README.md** - документация с первого дня
- 🎯 **.gitignore** - защита от случайной заливки лишнего
- 🎯 **Правильная последовательность** инициализации (Git → venv → README → commit)

**Технологии и инструменты:**
- Git для версионирования
- Python venv для изоляции backend
- npm/package.json для frontend (Node.js)
- README как точка входа для новичков
- .gitignore шаблоны для Python и Node.js

**Практические навыки:**
- Создание проекта в правильной последовательности
- Инициализация Git до написания кода
- Создание venv перед установкой пакетов
- Написание README с инструкциями установки
- Настройка .gitignore для защиты секретов
- Структурирование первого коммита

**Контекст применения:**
Пошаговый workflow инициализации. Показывает разницу между хаотичным подходом (сразу кодить) и правильным (Git → venv → README → commit). Учит как просить ИИ создавать воспроизводимую инициализацию.

#### **Урок 1.3: Dependencies - управление зависимостями**
**Основные концепции:**
- 🎯 **requirements.txt** (Python) - список зависимостей backend
- 🎯 **package.json** (Node.js) - список зависимостей frontend
- 🎯 **Semantic Versioning** - правила версионирования (major.minor.patch)
- 🎯 **Lock files** - фиксация точных версий (package-lock.json, pip freeze)
- 🎯 **Изоляция зависимостей** - у каждого проекта свои пакеты

**Технологии и инструменты:**
- pip freeze для Python зависимостей
- npm install для Node.js зависимостей
- Semantic versioning (^, ~, ==)
- Security audits (npm audit, safety)
- requirements.txt и package.json форматы

**Практические навыки:**
- Создание и обновление requirements.txt
- Понимание semantic versioning
- Воспроизведение окружения коллеги
- Проверка безопасности зависимостей (npm audit)
- Минимизация количества пакетов
- Обновление зависимостей безопасно

**Контекст применения:**
Аналогия с рецептом блюда. Показывает почему важно фиксировать версии для воспроизводимости. Учит работе с pip freeze, package.json, и когда обновлять зависимости.

#### **Урок 1.4: Virtual Environment (venv)**
**Основные концепции:**
- 🎯 **Изоляция зависимостей** - каждый проект в своей "песочнице"
- 🎯 **venv/ папка** - локальная копия Python с пакетами
- 🎯 **Активация/деактивация** - переключение между окружениями
- 🎯 **Конфликты версий** - как venv их предотвращает
- 🎯 **requirements.txt** - воспроизводимость окружения

**Технологии и инструменты:**
- `python -m venv venv` для создания
- `source venv/bin/activate` для активации (macOS/Linux)
- `venv\Scripts\activate` для Windows
- `pip install -r requirements.txt` для воспроизведения
- venv/ структура (bin/, lib/, include/)

**Практические навыки:**
- Создание виртуального окружения для каждого проекта
- Активация venv перед работой
- Установка пакетов только в активном venv
- Добавление venv/ в .gitignore
- Пересоздание venv при проблемах
- Проверка активации (which python, (venv) в терминале)

**Контекст применения:**
Аналогия с песочницами для детей. ASCII диаграммы показывают разницу между глобальной установкой и venv. Учит правильному workflow с ИИ для создания инструкций по venv в README.

#### **Урок 1.5: Environment Variables (.env)**
**Основные концепции:**
- 🎯 **Секреты отдельно от кода** - API ключи не в Git
- 🎯 **.env файл** - реальные секреты (скрытый dotfile, не в Git)
- 🎯 **.env.example** - шаблон без секретов (публичный, в Git)
- 🎯 **Dotfiles** - файлы с точкой скрыты от ИИ и людей
- 🎯 **Разные окружения** - dev/staging/production настройки

**Технологии и инструменты:**
- python-dotenv для Python (load_dotenv())
- dotenv или встроенная поддержка для Node.js
- NEXT_PUBLIC_ префикс для публичных переменных (Next.js)
- Секретные менеджеры для production (AWS Secrets, Vercel Variables)

**Практические навыки:**
- Создание .env.example (ИИ может создать)
- Создание .env вручную (ИИ НЕ может - dotfile)
- Заполнение реальных секретов самостоятельно
- Добавление .env в .gitignore
- Валидация обязательных переменных при старте
- Понимание NEXT_PUBLIC_ безопасности

**Контекст применения:**
Аналогия с ключами от квартиры. Объясняет почему ИИ не видит .env (dotfile) и зачем нужен .env.example. Показывает разницу между публичными (NEXT_PUBLIC_) и приватными переменными в Next.js.

#### **Урок 1.6: Git и .gitignore**
**Основные концепции:**
- 🎯 **Git как time machine** - история всех изменений кода
- 🎯 **Жизненный цикл файлов** - Working → Staging → Repository → Remote
- 🎯 **.gitignore** - защита от случайной заливки секретов и мусора
- 🎯 **Commit messages** - осмысленные сообщения (Add:, Fix:, Refactor:)
- 🎯 **Roadmap и Git связь** - задача → commit → отметка [X] в roadmap

**Технологии и инструменты:**
- git init, git add, git commit, git push
- .gitignore шаблоны для Python (venv/, __pycache__/) и Node.js (node_modules/)
- Semantic commit messages
- GitHub для облачного хранения

**Практические навыки:**
- Инициализация Git в начале проекта
- Создание правильного .gitignore (критичные: venv/, .env, node_modules/)
- Написание понятных commit messages
- Проверка что секреты не попали в Git (git status)
- Откат изменений при необходимости
- Восстановление секретов если случайно залил в Git

**Контекст применения:**
Аналогия с Google Docs версиями. ASCII диаграмма жизненного цикла файлов. Показывает что КРИТИЧНО в .gitignore. Учит связывать roadmap с commits. Объясняет emergency план если случайно залил секреты.

#### **Урок 1.7: Документация - правильная организация**
**Основные концепции:**
- 🎯 **Single Source of Truth (SSOT)** - каждая информация в одном месте
- 🎯 **README.md** - точка входа (50-150 строк), не энциклопедия
- 🎯 **docs/ папка** - детальная документация
- 🎯 **ADR (Architecture Decision Records)** - почему выбрали технологию X
- 🎯 **Три уровня документации** - простой/средний/большой проект

**Технологии и инструменты:**
- README.md как точка входа со ссылками
- docs/setup.md для детальной установки
- docs/architecture.md для схемы системы
- docs/decisions/ для ADR
- CHANGELOG.md для истории версий (Semantic Versioning)

**Практические навыки:**
- Создание компактного README (не раздувать до 1000 строк)
- Организация docs/ папки по назначению
- Использование ссылок вместо дублирования
- Написание ADR для важных решений
- Определение нужного уровня документации для проекта
- Обновление документации при изменениях кода

**Контекст применения:**
Показывает структуру docs/ для разных типов проектов (лендинг, full-stack, RAG). Объясняет SSOT принцип. Даёт шаблоны для README, ADR, CHANGELOG. Учит создавать ADR чтобы через год помнить решения.

#### **Урок 1.8: Планирование и Roadmap**
**Основные концепции:**
- 🎯 **Roadmap** - план проекта разбитый на этапы и задачи
- 🎯 **Маленькие задачи** - 1-3 часа работы для ИИ
- 🎯 **Итеративная разработка** - 3 итерации (прототип → MVP → production)
- 🎯 **Цикл** - Roadmap → Задача → Код → Commit → Обновление roadmap
- 🎯 **Конкретные измеримые задачи** - не "улучшить UI", а "добавить кнопку X"

**Технологии и инструменты:**
- roadmap.md с этапами и чекбоксами [ ] / [X]
- Markdown чеклисты для отслеживания
- Разбивка на 3-5 этапов
- Группировка по функционалу (не по технологиям)

**Практические навыки:**
- Создание roadmap перед началом кодинга (30-60 минут на планирование)
- Разбивка больших задач на маленькие (1-3 часа)
- Формулировка конкретных задач для ИИ
- Обновление roadmap после каждой задачи (отметка [X])
- Итеративная разработка (работающий прототип быстро)
- Работа по одной задаче (не перегружать контекст ИИ)

**Контекст применения:**
Аналогия со строительством дома (фундамент → стены → крыша). Показывает как roadmap предотвращает переполнение контекста ИИ. Даёт шаблоны roadmap для разных проектов (API, full-stack, RAG). Правило 1 часа: потратить 1 час на планирование = сэкономить 10 часов на рефакторинге.

#### **Урок 1.9: Рефакторинг - когда и как**
**Основные концепции:**
- 🎯 **Рефакторинг = уборка кода** - та же функциональность, лучше организовано
- 🎯 **Code Smells** - признаки грязного кода (дублирование, длинные функции)
- 🎯 **ПОСЛЕ того как работает** - сначала заставь работать, потом убирай
- 🎯 **Маленькими шагами** - по одному Code Smell за раз
- 🎯 **Make it work → Make it right → Make it fast**

**Технологии и инструменты:**
- DRY (Don't Repeat Yourself) принцип
- Разбивка длинных функций на маленькие
- Вынос бизнес-логики из routers в services
- Pydantic схемы для валидации (избегание дублирования)
- Git коммиты перед рефакторингом (для отката)

**Практические навыки:**
- Распознавание Code Smells (дублирование, функция >50 строк)
- Рефакторинг после того как фича работает
- Коммит перед рефакторингом (чтобы откатить)
- Проверка что всё работает после рефакторинга
- Работа с ИИ для рефакторинга конкретных проблем
- Знание когда НЕ делать рефакторинг (работающий понятный код)

**Контекст применения:**
Аналогия с уборкой квартиры - вещи те же, просто организованы лучше. Показывает примеры рефакторинга (убрать дублирование, разбить функцию, улучшить имена). Частота: ~1 раз в неделю или после 3-5 фич. Учит правильному workflow: работает → коммит → рефакторинг → проверка.
